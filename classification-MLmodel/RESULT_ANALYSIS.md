## ⚠️ 当前需要关注的问题

### 1. 24h 预测仍落后于朴素基线

- Persistence F1 达到 **0.5619**，优于所有 ML 模型（XGBoost 仅 0.5056）。
- 启用了增强特征后，短中期（1h/6h/12h）提升明显，但 24h 依旧缺乏长期信息。
- 建议：
  1. 进一步补充长周期滞后（48/72/168h）与周期编码，确保与 `split_indices` 对齐。
  2. 专门调参或采用多阶段建模应对 24h 的概念漂移。

### 2. MLP 仍是最弱模型

- Macro F1 仅 0.50 左右，尤其在 12h 只有 0.4319。
- 可能原因：搜索空间偏小、层数/正则不足，或标准化方式不适合当前特征规模。
- 建议：扩充 `hidden_layer_sizes`、`alpha`、`learning_rate_init` 的搜索范围，并延长 `n_iter`。

### 3. Horizon=24 的特征生成存在潜在异常

- `create_enhanced_features` 在 24h 分支直接引用 `co_lag24`，但函数内部没有定义该变量；若未来调用 `horizon=24` 会触发 `NameError`。
- 需要在函数顶部显式获取 `co_lag24 = co_lags.get(24)` 或类似逻辑后再引用。

### 性能退化模式

所有机器学习模型都随预测时间增长而性能下降：
- XGBoost: 0.73 → 0.58 → 0.53 → 0.51（-30%）
- Random Forest: 0.71 → 0.51 → 0.52 → 0.47（-34%）
- Logistic Regression: 0.72 → 0.50 → 0.46 → 0.44（-39%）

但Persistence Baseline在24h表现相对稳定（0.73 → 0.56，-23%），说明：
- 长期预测中，简单的"当前值"策略可能更有效
- ML模型需要更好的长期依赖特征

## 🔧 改进建议

1. **针对 24h 构造更强的长期特征**：落地 `48/72/168h` 滞后、周序列编码，并校验索引对齐，避免引入 NaN。
2. **修复 `co_lag24` 未定义问题**：确保 `create_enhanced_features(..., horizon=24)` 时不会抛错，便于后续试验。
3. **增强 MLP 搜索空间**：适当提高层数、节点数及迭代次数，必要时加入 `BatchNorm` 或 Dropout。
4. **记录当前实验设定**：在 Notebook 中打印“增强后特征数”“缺失值填充策略”，方便将来比对。
